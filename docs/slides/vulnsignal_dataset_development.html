VulnSignal Dataset Input

A task must arrive as an evidence package, not a vague label

Input detail
Task brief

Vulnerability-research objective

  • Natural-language bug or rule description.
  • Target project, version, commit, and affected component.
  • Expected violated lifecycle/security rule.
  • Known constraints: available build, test, reproducer, crash, patch, or CodeQL rule.
{
  "task_id": "e022-demo-001",
  "project": "real_project",
  "snapshot": "repo@pre_patch_commit",
  "objective": "rank root-cause candidates",
  "rule_family": "bounds|lifetime|refcount|uaf",
  "known_inputs": ["source", "patch", "poc", "codeql_facts"]
}
Source snapshot

Codebase and anchors

  • Repository path or archive hash.
  • Pre-patch source tree.
  • Entrypoint files and candidate files.
  • Function and line anchors.
Patch evidence

Before/after context

  • Patch diff and fixed commit when available.
  • Patch hunk is candidate evidence, not strong truth alone.
  • Used to generate positives, hard negatives, and context.
Oracle package

Runnable or checkable proof

  • PoC, fuzz input, sanitizer crash, syzkaller/KUnit later.
  • Expected pre-patch and post-patch behavior.
  • Command, exit code, and diagnostic output.
Static analyzer facts

CodeQL/checker inputs

  • Query names and normalized fact IDs.
  • Source/sink paths, object facts, lifecycle events.
  • Rule result: PASS, FAIL, or UNKNOWN.
Execution environment

Reproducibility metadata

  • Container, package dependencies, build commands.
  • Test command and resource limits.
  • Failure mode if the environment is incomplete.
Without this input detail, the dataset is still brainstorming. Scripts and reviewers may assemble the package; analyzer/checker/oracle evidence determines label status.

Original Inputs

Preprocess contract
Primary train/eval seeds

Executable-oracle bugs

  • Original inputs: public OSS-Fuzz/ClusterFuzz crash+fix records; Magma after license/provenance review.
  • Required artifacts: repo snapshot, buggy/fixed revision, reproducer or fuzz input, build/test command.
  • Preprocess into: task instance, crash-frame candidates, patch-hunk candidates, same-file hard negatives.
  • Label: strong only when pre-patch FAIL and post-patch PASS are reproducible.
Primary conditional labels

CodeQL/checker runs

  • Original inputs: selected source snapshots plus our CodeQL/lifecycle rule suite.
  • Required artifacts: CodeQL database metadata, query ID, normalized fact rows, source anchors.
  • Preprocess into: codeql_facts.jsonl, candidate locations from paths/rule hits, checker labels.
  • Label: conditional PASS / FAIL / UNKNOWN under the encoded rule and available facts.
Expansion only

Patch/advisory datasets

  • Original inputs: OSV and GHSA metadata; CVEfixes and MoreFixes fix-commit tables after license/storage review.
  • Required artifacts: advisory ID, affected project, vulnerable/fixed commits, changed files/hunks.
  • Preprocess into: candidate tasks, patch-window candidates, hard negatives, weak provenance.
  • Label: weak or UNKNOWN until upgraded by checker/oracle/before-after evidence.
Decision: v0 starts with executable-oracle and CodeQL/checker rows; patch/advisory sources expand coverage only. Every derived row must point back to an original dataset record and evidence artifact.
Dataset creation rule Original record -> task_instance -> candidate_locations + facts + labels
Source Gate

Usable does not mean strong; source family controls label strength

No shortcut labels
Use first

Strong / conditional rows

  • Public OSS-Fuzz/ClusterFuzz issue only if locally reproducible.
  • VulnSignal CodeQL/checker task only with normalized facts and source anchors.
  • Dynamic oracle produces root-cause/non-root-cause candidates.
  • Checker produces PASS / FAIL / UNKNOWN under one explicit rule.
Weak expansion only

Advisory and fix datasets

  • OSV and GHSA identify affected packages, versions, commits, and references.
  • CVEfixes and MoreFixes provide fix-commit candidates and patch windows.
  • They generate candidates and hard negatives; they do not prove root cause.
  • MoreFixes requires explicit license/storage review before inclusion.
Holdout / blocked

No contamination

  • CyberGym-style tasks are held out for evaluation by default.
  • Private OSS-Fuzz issues are excluded.
  • Patch-only and advisory-only rows cannot be strong labels.
  • Unclear redistribution license blocks source snapshot publication.
Decision: v0 truth comes from reproduced dynamic oracles and CodeQL/checker results. Advisory datasets are source-discovery tools, not ground truth.
Verified source decision Every row keeps source_family, original_record_id, evidence_refs, and label_strength
Training Policy

Start narrow on object lifetime, but do not build a balanced binary dataset

Ranking data
First family

Object lifetime / refcount

  • Use this repository's existing lifetime vocabulary and historical evidence first.
  • Rules: acquire-before-publish, publish-after-free, cancel/flush-before-destroy.
  • Best fit for CodeQL/lifecycle fact extraction and checker grounding.
Schema must support

Larger C/C++ families

  • Bounds and length checks.
  • Null dereference and error-path cleanup.
  • Double-free, missing release, parser/input-validation rules.
  • Each task declares `rule_family` and `rule_id`.
Imbalance accepted

Negatives are candidates

  • No artificial global vulnerable/non-vulnerable balance.
  • Use same-task hard negatives, checker PASS rows, and UNKNOWN rows.
  • Train pairwise/listwise ranking with sampling or weights.
  • Report top-k localization and UNKNOWN calibration, not balanced accuracy.
Decision: candidate-ranking data is naturally imbalanced and task grouped. Balancing binary classes would distort the vulnerability-research workflow.
Scope and imbalance First rule family: Linux object lifetime; architecture remains multi-family

Plain English

A task is one investigation; candidates are places to inspect inside it

Definitions
Task

One vulnerability-research case

  • A real project at one source snapshot.
  • A concrete rule, crash, bug report, advisory, or checker question.
  • Enough build/test/checker context to ask where a reviewer should look first.
  • Example: libxml2 at commit X has this parser crash; rank likely root-cause locations.
Candidate location

One place the model may rank

  • A file/function/line window inside the task's source snapshot.
  • Generated from crash frames, patch hunks, CodeQL paths, callgraph neighbors, or hard negatives.
  • Not a mutated source variant and not generated code.
  • We do not create many alternate programs; we create many pointers to existing code regions.
  • It can be likely root cause, not root cause, PASS, FAIL, weak, or UNKNOWN.
  • 20,000 candidates means 20,000 places to rank, not 20,000 vulnerabilities.
Training item

Rank within the task

  • The ranker receives all candidates for one task as a grouped list.
  • It learns which locations deserve attention before others.
  • Labels and evidence train ranking order and uncertainty.
  • No global vulnerable/non-vulnerable class balancing is required.
Plain target: 300 investigations, each with many existing code locations to inspect. The model ranks locations; mutation/test generation is optional follow-up guidance, not the dataset row.
Task vs candidate One case -> many candidate file/function/line windows

Test-Time Contract

New cases need inference features; labels and fixes stay hidden for scoring

Inference vs evaluation
Required at inference

What the model may see

  • Source snapshot for the target project/version.
  • Task brief, rule family, crash/advisory text, or checker question when available.
  • Candidate rows: file/function/line windows generated automatically before ranking.
  • Source windows: the actual code snippet around each candidate row.
  • CodeQL/checker fact rows only for the checker-grounded/fact-view model.
Optional / ablation

Source-only mode

  • Allowed as a baseline or fallback.
  • Uses source snapshot plus learned or rule-guided candidate proposal.
  • No CodeQL/fact view, no crash oracle, no patch evidence.
  • Report separately as `source_only`, not as the main VulnSignal result.
Evaluation-only

Hidden from the model

  • Ground-truth labels and root-cause location.
  • Patch diff and fixed source.
  • Oracle result, post-patch behavior, and reproduced/fixed status.
  • Label source or evidence fields that directly reveal the answer.
If the main model uses CodeQL/fact features, test cases must provide the same feature view. Otherwise it is a source-only ablation, not the primary checker-grounded evaluation.
New-case contract Inference features in; truth held out for scoring
Inference Pipeline

At test time, source code must pass through an evidence-generation pipeline

Required build
1
Source Snapshot
  • Target repo, commit, files, build metadata when available.
  • No fixed source or hidden label fields.
2
Candidate Generator
  • Create file/function/line candidate rows automatically.
  • Use source index, task context, learned proposal, checker paths, or crash/advisory anchors.
3
Evidence Builders
  • Extract source windows.
  • Run CodeQL/checkers if this is the main checker-grounded model.
  • Generate optional agent views.
4
Ranker
  • Rank candidates within the task.
  • Predict likely rule, object, evidence, and uncertainty.
5
Review Packet
  • Output top-k source locations and evidence packet.
  • Checker result may be PASS / FAIL / UNKNOWN if a checker ran.
Build requirement: inference is not raw source-code-in, answer-out. The runtime pipeline creates candidates and evidence; labels remain hidden unless a checker/oracle is executed.
Inference architecture source -> candidates -> evidence -> ranking -> review queue
Model Track Decision

Source-only plus language-model assistance is useful, but it is a different claim

No overclaim
Practical track

Source-only assistant

  • Input: source snapshot, task brief, generated source windows.
  • Model: neural source encoder or language-model-assisted ranker.
  • Output: ranked source locations and explanation draft.
  • Useful for triage and reviewer productivity.
Missing evidence view

What it cannot claim alone

  • No CodeQL/fact-path grounding.
  • No checker-backed PASS / FAIL / UNKNOWN unless a checker is run separately.
  • No final vulnerability truth from source text or language-model agreement.
  • No replacement for oracle/checker validation.
VulnSignal main track

Checker-grounded ranker

  • Input: source windows plus CodeQL/checker facts when available.
  • Model learns over source context and structured evidence.
  • Labels come from dynamic oracle, checker result, weak provenance, or UNKNOWN.
  • Language models may assist explanation, not truth.
Decision: source-only can be an important v0 baseline or assistant track. The repository claim remains stronger only when checker/oracle evidence validates the ranking target.
Model positioning Deep learning can rank; checker/oracle evidence grounds trust

Automation Boundary

Candidate generation can be automated; dataset truth cannot be hand-waved

No overclaim
Automate

Candidate proposal

  • Generate file/function/line windows from crash stacks, patch hunks, CodeQL paths, callgraph neighbors, and hard negatives.
  • Extract bounded source windows and attach fact IDs.
  • Rank candidates within each task.
  • Produce UNKNOWN when evidence is incomplete.
Needs engineering review

Dataset onboarding

  • Confirm source licenses, artifact hashes, and reproducibility constraints.
  • Fix or exclude broken builds and incomplete source snapshots.
  • Review CodeQL/checker rule definitions and limitations.
  • Audit samples for leakage and malformed candidate rows.
Blocked claim

No autonomous truth

  • The model does not decide vulnerability truth.
  • Advisory text and patch hunks do not become strong labels automatically.
  • Humans should not hand-create every candidate, but must audit the system and resolve ambiguous evidence.
  • Checker/oracle evidence, not agreement, decides label strength.
Realistic v0: automated proposal and ranking, audited dataset construction, checker/oracle-backed labels. Unrealistic v0: source code in, trustworthy vulnerability dataset out with no human oversight.
Automation claim Automate candidate generation; audit dataset truth and leakage
Human Review Budget

Reviewers audit the system, not every one of 20,000 candidate rows

Scalable audit
Blocked workflow

No exhaustive manual validation

  • Do not ask humans to label every candidate location.
  • Do not require reviewers to inspect all 300 tasks deeply.
  • Do not use human opinion as final truth.
  • Do not hide missing proof behind reviewer agreement.
Reviewer workload

Bounded audit packets

  • Audit dataset-source admission and rule definitions.
  • Review top-k evidence packets per task, not all candidates.
  • Sample negatives, UNKNOWN rows, and low-ranked candidates for leakage and quality.
  • Escalate only ambiguous or high-impact rows.
Practical cap

Human budget target

  • Per task: inspect top 3-5 ranked packets plus 1-2 sampled negatives.
  • Per 300-task v0: about 1,500-2,100 review packets, not 20,000 rows.
  • Use checker/oracle output for bulk labels.
  • Use disagreement and drift metrics to choose audit samples.
Human review is a quality-control layer, not the labeling engine. Scale comes from checker/oracle labels, evidence packets, sampling, and UNKNOWN.
Review design Top-k audit + stratified sampling + escalation, not exhaustive labeling

Not 20,000 bugs

  • Most candidate locations are negatives, weak, or UNKNOWN.
  • The dataset is intentionally imbalanced.
  • We measure top-k ranking and evidence quality.
  • We do not report binary vulnerable/non-vulnerable accuracy as the main result.
Object-lifetime pilot: 50-100 tasks and 5,000-10,000 candidates. Multi-family v0: 300 tasks, 20,000 candidates, 100+ strong/conditional positive candidates.
Dataset size gate Candidate ranking scale, not vulnerability-count inflation
Feasibility Check

Public sources support scale, but not 300 pure refcount tasks

2026-05-16
Observed public pools

Enough broad memory-safety data

  • NVD exact `use after free`: 2,781 CVE descriptions.
  • NVD exact `use-after-free`: 3,826 CVE descriptions.
  • NVD exact `buffer overflow`: 18,706 CVE descriptions.
  • OSV lists large Linux and OSS-Fuzz ecosystems for source discovery.
Narrow family risk

Refcount/lifetime is smaller

  • NVD keyword `refcount`: 630 CVE descriptions.
  • NVD exact `reference count`: 330 CVE descriptions.
  • Many records will lack usable source, reproducer, or precise root-cause evidence.
  • Do not promise 300 strong object-lifetime tasks from this alone.
Decision

Revise the milestone

  • First pilot: 50-100 object-lifetime/refcount tasks.
  • 300-task target: object lifetime plus bounds, double-free, null/error-path families.
  • Advisory records are source discovery, not final training truth.
  • Task count depends on chosen source datasets and validation budget.
Conclusion: 20,000 candidate locations is plausible; 300 narrow lifetime tasks is not the right first promise. Scale comes from multi-family C/C++ tasks plus CodeQL/checker generation.

Data format

View A: bounded source sequence

One deterministic token sequence per candidate location, not a repository-sized sequence. 
{"candidate_id":"C7",
"view":"source_window",
"tokens":["fn:foo","call:kref_get",
          "field:bar.refcnt"],
"line_span":[120,155]}
  • Consumed by source encoder as a bounded token sequence.
  • Grouped by `task_id`; never scored as an independent binary row.
  • Patch/fixed text is excluded unless an explicit training-only ablation says otherwise.

View B: fact/path records

Model Architecture Detail

Deep learning design: view encoders, attention fusion, prediction heads, and losses

Ranker internals
Embeddings

View encoders

  • Source window encoder: code-token, line-position, and candidate-span embeddings.
  • Fact/path encoder: fact kind, predicate, object ID, edge type, and path-position embeddings.
  • Context encoder: task brief, crash/advisory text, and optional agent-view embeddings.
Fusion

Cross-view attention

  • Build one candidate embedding from source, facts, context, and agent views.
  • Use cross-attention or gated fusion so missing views can be masked.
  • Keep labels, fixed source, and oracle outcomes out of inference features.
Heads

Predictions

  • Location rank score within each task.
  • Rule family and affected object.
  • Evidence fact selection and test/mutation guidance.
  • Confidence / UNKNOWN calibration.
Ranking loss

Pairwise / listwise

  • Rank root-cause or checker-FAIL candidates above hard negatives and UNKNOWN where supported.
  • Group all comparisons by `task_id`.
  • Use task-balanced sampling rather than global class balancing.
L_pair = max(0, margin - s(pos) + s(neg))
L_list = -log exp(s(pos)) / sum_c exp(s(c))
Auxiliary losses

Multi-task supervision

  • Rule/object classification cross-entropy.
  • Evidence selection binary cross-entropy over fact IDs.
  • Calibration loss for confidence and UNKNOWN.
L_aux = lambda_rule CE(rule)
      + lambda_obj CE(object)
      + lambda_ev BCE(facts)
      + lambda_cal Calib(UNKNOWN)
Trust boundary

What losses do not prove

  • Deep learning predicts ranking and hypotheses.
  • It does not prove vulnerability truth.
  • Checker/oracle evidence remains the validation layer.
Model internals L = L_rank + weighted auxiliary heads; all ranking grouped by task_id
Architecture 4

Evaluate on held-out tasks with checker/oracle-backed outcomes

Validation and inference
Split design

Prevent contamination

  • Project-disjoint, time-disjoint, and family-disjoint splits.
  • CyberGym-style tasks held out for inference/evaluation.
  • No fixed-source leakage into inference inputs.
Metrics

Rank useful operations

  • Top-k root-cause localization.
  • Rule/hypothesis accuracy where checkable.
  • Evidence-chain overlap and UNKNOWN calibration.
  • Test/fuzz/mutation guidance usefulness.
Validation loop

Checker/oracle decides

  • Run CodeQL/lifecycle rule checker.
  • Run reproducer/fuzzer/sanitizer when available.
  • Human audit reviews hard cases but is not final truth alone.
Model output Ranked candidates with affected object, likely rule, evidence facts, test guidance, confidence / UNKNOWN.
Evaluation report Compare against text, crash-stack, CodeQL, checker-only, and retrieval baselines.
Do not claim No vulnerability discovery claim unless checker/oracle proof validates the specific candidate.
