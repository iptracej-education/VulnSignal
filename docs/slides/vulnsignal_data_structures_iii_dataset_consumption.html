Data Structures III: Dataset Consumption

Training consumes candidate sets grouped by task instance. Evaluation checks whether the ranker proposes useful vulnerability-research operations under strict leakage controls.

  • Ranking is within a task, not across unrelated repositories.
  • UNKNOWN is a first-class outcome when facts are insufficient.

Training Objective

  • Pairwise or listwise candidate ranking within each task_id.
  • Higher rank for candidates supported by dynamic oracle or CodeQL/checker evidence.
  • Weak patch/advisory labels can guide pretraining or analysis, not final truth claims.

Baselines Required

  • Random candidate ranker.
  • Crash-stack or patch-proximity heuristic.
  • Source-text only and API-token only rankers.
  • CodeQL path/checker-only ranking.

Evaluation Split Policy

  • Held-out by project, time, source family, and rule family where possible.
  • CyberGym-style tasks are held-out inference/evaluation, not contaminated training.
  • No fixed-source, post-patch, or label-source leakage into model inputs.

Blocked Claims

  • Do not claim vulnerability discovery without checker/oracle proof.
  • Do not frame the model as a generic vulnerable / non-vulnerable classifier.
  • Do not use LLM consensus or human opinion as ground truth.
  • Do not make reward modeling the immediate next step.
